# Guide

This guide walks you step-by-step through configuring and provisioning an Azure Virtual Machine, installing Docker inside the VM guest OS, containerizing a Python FastAPI server running inside the VM, and testing the endpoint security via OAuth client credentials tokens.

---

## 1. Zero-Trust Access Control with JWT

In a secure cloud environment like **RetailBank-Pro**, APIs are protected by validating JSON Web Tokens (JWT) issued by Microsoft Entra ID. The client requests a token via OAuth 2.0 Client Credentials flow, then attaches it to request headers:

```
[ Client Gateway ] -- 1. POST /token --> [ Entra ID Identity Provider ]
[ Client Gateway ] <- 2. Return JWT ---- [ Entra ID Identity Provider ]

[ Client Gateway ] -- 3. POST /transactions (Header: Authorization: Bearer <JWT>) --> [ Transaction API Container ]
```

When a request arrives at the Transaction API, it validates:

1. **Issuer (`iss`):** The token was generated by the correct tenant.
2. **Audience (`aud`):** The token was intended for this API scope.
3. **Signature:** The token has not been tampered with.
4. **App Role (`roles`):** The client has been granted the required permission (`Transaction.Write`).

## 2. Lab Steps: VM Provisioning and Security Setup

### Step 2.1: Provision the Azure VM Infrastructure

Run these commands inside your local SRE terminal (such as WSL, Git Bash, or Azure Cloud Shell) to create the hosting infrastructure.

```bash
# 1. Create a Resource Group
az group create --name rg-retailbank-prod --location eastus

# 2. Create the Network Security Group (NSG)
az network nsg create --resource-group rg-retailbank-prod --name nsg-retailbank-prod

# 3. Configure NSG inbound security rules for SSH (22) and FastAPI (8000)
az network nsg rule create -g rg-retailbank-prod --nsg-name nsg-retailbank-prod -n Allow-SSH-Inbound --priority 100 --destination-port-ranges 22 --access Allow --protocol Tcp
az network nsg rule create -g rg-retailbank-prod --nsg-name nsg-retailbank-prod -n Allow-FastAPI-8000 --priority 110 --destination-port-ranges 8000 --access Allow --protocol Tcp

# 4. Deploy host VM (Ubuntu 22.04, size Standard_B1s for cost control)
az vm create \
  --resource-group rg-retailbank-prod \
  --name vm-retailbank-host-01 \
  --image Ubuntu2204 \
  --size Standard_B1s \
  --admin-username azureuser \
  --nsg nsg-retailbank-prod \
  --generate-ssh-keys
```

Make a note of the VM's Public IP address returned by the `az vm create` output.

## 3. Lab Steps: Configuring Guest OS and Deploying Docker Container

### Step 3.1: Connect and Install Docker

SSH into the newly deployed virtual machine:

```bash
ssh azureuser@<VM-PUBLIC-IP>
```

Update packages and install Docker:

```bash
sudo apt-get update && sudo apt-get install -y docker.io
sudo systemctl start docker
sudo systemctl enable docker

# Configure user permissions to run docker without sudo
sudo usermod -aG docker azureuser
newgrp docker
```

### Step 3.2: Deploy the Application Code & Dockerfile

Create a `code` directory on the VM host:

```bash
mkdir -p ~/code && cd ~/code
```

Create `demo_aad_auth_app.py` inside the VM and copy Monday's FastAPI code into it:

```bash
nano demo_aad_auth_app.py
```

*(Paste the FastAPI script content, then save and exit using Ctrl+O, Enter, and Ctrl+X).*

Create Monday's `Dockerfile` in the same directory:

```bash
nano Dockerfile
```

*(Paste the Dockerfile content, then save and exit).*

### Step 3.3: Build and Run the Container

Build the Docker image from source and launch the containerized application. Binding port 8000 exposes the FastAPI endpoint outside the container:

```bash
# Build the Docker image
docker build -t retailbank-auth-api .

# Run the container mapping host port 8000
docker run -d --name auth-api-service -p 8000:8000 retailbank-auth-api
```

## 4. Lab Steps: Running the Client Verification Tests

On your local workstation terminal, review `code/demo_aad_auth_flow.sh`. This script can build the Docker container locally or test a remote VM endpoint.

Execute the test suite locally:

```bash
bash code/demo_aad_auth_flow.sh
```

Execute the test suite targeting your deployed Azure VM:

```bash
bash code/demo_aad_auth_flow.sh <VM-PUBLIC-IP>
```

Review the response logs and ensure they match the expected codes:

- **Test A:** No authorization token -> Returns **`401 Unauthorized`**.
- **Test B:** Malformed token -> Returns **`401 Unauthorized`**.
- **Test C:** Valid token but missing the `Transaction.Write` role -> Expects **`403 Forbidden`**.
- **Test D:** Valid token with correct roles -> Expects **`200 OK`**.

## 5. Reference Troubleshooting Table

| Issue / Symptom | Cause | Mitigation |
| --- | --- | --- |
| `Permission denied` when running Docker commands. | Current terminal session has not reloaded user group permissions. | Run `newgrp docker` or log out and log back in via SSH. |
| `Connection refused` when curling port 8000. | Container is not running, or port forwarding is misconfigured. | Run `docker ps` to verify the container is active and ports show `0.0.0.0:8000->8000/tcp`. |
| `Connection timeout` when curling from local machine. | Azure NSG rule on port 8000 is missing or misconfigured. | Verify that you ran `az network nsg rule create` for port 8000 on the VM. |
